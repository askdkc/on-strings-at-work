# On Strings at Work

## インターンのミナはEmacsで文字列地獄をほどく

この本は、Emacs好きのミナが、会社の汚いメール、CSV、ログ、JSONを整理し、最後にはPostgreSQLへ流し込める形に変えていく物語です。

ミナが最初にするのは、コードを書くことではありません。  
まず、何が混ざっているのかを見ます。  
次に、それを言葉にします。  
そして、行や項目のリストへ分けていきます。

一般的な仕事の多くは、日々発生するデータとの向き合い方に収束していきます。メール、表、ログ、APIレスポンス、申請データ、顧客情報。形は違っても、そこには「読みにくい」「揺れている」「機械に渡しにくい」情報があります。

この物語を、日々の業務を処理しやすく分解し、少しずつ自動化へ向かわせるためのガイドとして使ってください。

難しい話は、ミナの仕事を追いながら少しずつ出てきます。

## 1. インターンのミナ、会社へ行く

水野ミナは、大学を卒業した。

卒業はした。けれど、就職活動には最後まで身が入らなかった。ゼミの友人たちがスーツを着て説明会へ行っている間、ミナは夜中にEmacsの設定をいじっていた。卒論の体裁を整えるために入れたはずのEmacsは、いつの間にか彼女の生活の中心になっていた。

Emacsで講義ノートを取り、レポートの変更履歴を見比べ、大学のサーバーに入ってファイルを直していた。便利にするための設定ファイル、つまり `init.el` はどんどん長くなった。補完を速くしたい。見た目を整えたい。キー操作を自分の手に合わせたい。そうやって一晩中いじっているうちに、肝心の課題は締切ぎりぎりになり、単位も危うくなった。

卒業式の日、友人に言われた。

「ミナ、就職どうすんの？」

ミナは笑ってごまかした。

「まぁ、なんとかなるっしょ。」

なんとかは、ならなかった。

卒業後、ミナは実家に戻った。昼前に起きて、ThinkPadを開き、Arch Linuxを更新し、Hyprlandの設定をいじり、Emacsをソースからコンパイルし直し、気づくと夕方になっていた。

母は最初こそ何も言わなかったが、三か月目に入ると、さすがに声が鋭くなった。

「ミナ、あなた、毎日パソコンで何してるの？」

「環境育ててる。」

「仕事は？」

「うっ。」

そんなある日、父が夕食の席で言った。

「知り合いの会社で、パソコンに詳しい人を探してるらしい。」

ミナは箸を止めた。

「パソコンに詳しい人って、だいたい雑用係にされるやつじゃん。」

「まあ、最初はそうかもしれないな。」

父は悪びれずに続けた。

「黒瀬さんっていう昔の友人がいてね。中小企業の情報システム部を見ている。社内にコンピューターに詳しい若い人がいないらしい。正社員じゃなくても、インターン感覚でいいから来てみないか、って。」

ミナは少し黙った。

就職活動に失敗したわけではない。そもそも、真面目にやっていなかった。だから、胸を張れる状況ではなかった。けれど、何もしないまま実家でEmacsを育て続けるのも、そろそろ気まずい。

「……週何日？」

「最初は週三日でもいいらしい。」

「ThinkPad持って行っていい？」

「それは聞いてみればいい。」

「じゃあ、インターン感覚なら行く。合わなかったら帰る。」

父は笑った。

「それでいい。まず行ってみなさい。」

一週間後、ミナは黒いリボンのついたブラウスに、少し派手なネイル、毛先にピンクのメッシュを入れた髪で、その会社の受付に立っていた。肩にはThinkPadの入ったバッグ。天板には、青い三角形のステッカーが貼ってある。

会社の名は、桐和システムサービス。

大企業ではない。だが、営業部、総務部、経理部、開発チーム、情報システム部が一通りあり、古い業務と新しいクラウドサービスが微妙な形で混ざっている会社だった。

受付の奥から、少し大柄な男性が出てきた。

「水野ミナさん？」

「はい。」

「黒瀬です。お父さんには昔から世話になっていてね。今日は来てくれてありがとう。まあ、堅い面接じゃないから、気楽にして。」

黒瀬は情報システム部の課長だった。人当たりは柔らかいが、目の奥は疲れている。社内の面倒なものが全部ここに流れてくる部署の人間の目だった。

ミナが案内された小さな会議室には、もう一人、細身の男性がいた。

「こっちは高村。うちの情シスの先輩社員。ネットワークも端末管理も、だいたい高村が見てる。」

高村は軽く手を上げた。

「よろしく。今日は見学くらいのつもりでいいよ。黒瀬課長が『コンピューターに詳しい子が来る』って言うから、どんな子かと思ってた。」

ミナは軽く会釈した。

「よろしくお願いします。詳しいっていうか、Emacsが好きなだけです。」

高村は一瞬だけ目を細めた。

「Emacsって、あの古いエディタのEmacs？」

その反応は、知らない人の反応ではなかった。知ってはいるが、今でもそれを主力で使う人間に少し驚いている顔だった。ミナは少しだけ安心した。

この時点では、まだ誰も知らなかった。  
このインターンが、社内の文字列地獄を少しずつほどいていくことになるとは。

### ミナのメモ

ミナは右下のメモバッファに、ここまでの状況を短く整理した。

```text
現実:
  ミナは正社員ではなく、父の紹介でインターン感覚で会社に来た。

人物:
  黒瀬 = 情報システム部の課長。業務全体を見る人。
  高村 = 情シスの先輩社員。技術側の現実を知る人。
  ミナ = 社会人経験は薄いが、Emacsと文字列には妙に強い人。
```

## 2. ミナの環境――ThinkPad、Arch Linux、Hyprland、Emacs

会議室の机に、ミナはThinkPadを置いた。

天板には小さな傷がいくつかあった。角にも少し擦れがある。新品のきらきらしたノートPCではない。けれど、ミナの手つきは雑ではなかった。バッグから出す時も、机に置く時も、慣れた道具を扱うように自然だった。

黒瀬が言った。

「そのノート、けっこう使い込んでるね。」

「ThinkPadはそこがいいんです。壊れにくいし、キーボードが打ちやすいし、変に気を使わなくていい。かわいい薄型PCも好きですけど、毎日使うなら頼れる相棒感が大事です。」

高村が頷いた。

「昔から、Linux入れて使う人も多い印象はあるね。」

「はい。あと、ディスプレイがかなり開くのも地味に便利です。」

黒瀬は首を傾げた。

「そんなに開いて、何が便利なの？」

ミナはThinkPadの画面をぐっと倒し、机の上でかなり低い角度にした。

「会議室で向かいの人に画面を見せる時、膝の上で作業する時、机が狭くて姿勢が変な時、角度の自由が効くんです。派手じゃないけど、地味に効きます。こういう地味な便利さ、長く使うと大事なんですよ。」

黒瀬は少し笑った。

「なるほど。スペック表に出にくい良さだ。」

「そうです。毎日触る道具は、そういうところが効きます。」

黒瀬は天板の青い三角形のステッカーを見て、首を傾げた。

「そのAみたいなステッカーは何？」

「Arch Linuxです。Linuxの一種です。」

「普通のWindowsとかMacとは違うの？」

「違います。Linuxは、パソコンの中身を自分で組み立てやすいOSです。その中でもArchは、最初から全部そろった弁当じゃなくて、自分で具を選んで詰める弁当箱みたいな感じです。」

「自由だけど面倒そうだね。」

「面倒です。でも、自分が何を入れたか分かるので、壊れたときに追いやすいです。」

黒瀬は笑った。

「面倒臭がりなのに、面倒なものを使うんだ。」

「普段の作業を楽にするための面倒は、別腹です。」

高村が画面を覗き込んだ。

「デスクトップも普通と違うね。」

「Hyprlandです。画面をタイル状に並べる環境です。ウィンドウを重ねて探すんじゃなくて、最初から左、右上、右下みたいに場所を決めて置きます。」

「机の上に紙を並べる感じか。」

「そうです。元データ、端末、メモを見える場所に置いて、考えが迷子にならないようにします。」

高村は画面を見ながら言った。

「Emacsって、あの古いエディタのEmacs？」

「そうです。」

「今どきならNeoVimを使う人も多いけど、そっちじゃないんだ。」

高村は黒瀬に説明した。

「NeoVimは、軽くて速くて、端末で使う人が多いエディタですね。Emacsと比べられがちです。」

黒瀬はミナを見た。

「じゃあ、なんでNeoVimじゃなくてEmacs？」

ミナは少しだけ目を丸くしたあと、ふふっと笑った。

「ふふふっ！ 盛るのが楽しいからですよ！」

「盛る？」

「はい。メイクも服も、手間だからやめるんじゃなくて、かわいくなるからやるんです。環境も同じです。自分が毎日触るものなら、かわいく、使いやすく、気分が上がるように育てたいじゃないですか。」

高村が吹き出した。

「今回来たインターン、自己紹介が濃いな。」

ミナは肩をすくめた。

「NeoVimも強いです。でも私にとってEmacsは、文章を書く場所でもあるし、ログを見る場所でもあるし、Lispを試す場所でもあります。編集環境の中心なんです。」

「つまり、エディタというより、仕事で使う道具をまとめて置ける環境なんだね。」

「そうです。だからソースから入れてます。どんなオプションで動いてるか、自分で把握しておきたいので。」

ミナはEmacsを全画面で開いた。HyprlandのワークスペースいっぱいにEmacsが広がる。左のウィンドウにはテキストを貼るバッファ、右上にはeatモードのターミナル、右下には抽出結果とメモ用のバッファを置いた。

eatモードは、Emacsの中でシェルやコマンドを動かすための端末画面である。

```text
Hyprland workspace
  └─ Emacs full-screen
       ├─ 左    : 元データやメール本文のバッファ
       ├─ 右上  : eatモードのターミナル
       └─ 右下  : 抽出結果、作業メモ、差分確認用バッファ
```

黒瀬は画面を見た。

「これ、いくつアプリを使ってるの？」

「今はEmacs一個です。」

「メール本文を見て、端末を使って、メモも取るのに？」

「はい。見る場所と、試す場所と、記録する場所が離れないのがいいんです。」

高村は少しだけ感心したように言った。

「それは確かに面白いな。古いエディタってだけじゃないわけだ。」

### ミナのメモ

ミナは右下のメモバッファに、ここまでの状況を短く整理した。

```text
ThinkPad:
  ミナのノートPC。

Arch Linux:
  自由度が高いが、組み立てる手間もあるLinux。

Hyprland:
  画面をタイル状に並べる環境。

Emacs:
  文章やコードを書くエディタ。
  ミナにとっては、テキスト、端末、Lisp実験、メモを一つの編集環境に置ける中心。

この章の意味:
  ミナがあとで文字列地獄を処理できる理由を、先に読者へ見せる。
```

## 3. 最初の文字列地獄

自己紹介が一段落したところで、黒瀬は一枚の紙をミナに渡した。

```text
インターン初日 課題1:
営業部から来た申請メールを整理し、週次でDB投入できる形にすること。
```

ミナは少しだけ安心した。

「メール整理なら、最初の仕事としては普通かもしれない。」

そう思ったのは、最初の五分だけだった。

黒瀬が説明した。

「営業部から申請メールが来る。今は担当者が一件ずつ開いて、必要な項目をExcelへコピペしている。金曜にそのExcelをCSVにして、社内DBへ流し込む。」

「毎週ですか？」

「毎週。」

「件数は？」

「少ない週で百件。多いと千件近い。」

ミナは口元を引きつらせた。

「うげっ、それ手でやるんすか。ダルぃなぁ……。」

黒瀬は声を出して笑った。

「ははははっ。いや、その反応は正しい。インターン初日の仕事としては、業務の流れを知るには悪くないんだけどね。」

「流れを見る前に心が死にそうですけど。」

「そこまで言う？」

最初のメールは、まだ読めた。

```text
件名: 申請 A-1029
氏名: 山田　太郎
部署: 営業一課
金額: 12,000円
希望日: 2024/04/01
備考: 急ぎ
```

次のメールは、少し違った。

```text
申請番号 A-1030
名前： 佐藤 花子
部門：営業2課
金額：12000
希望日：2024年4月2日
```

さらに別のメールは、長い挨拶文の中に必要情報が埋もれていた。

```text
お世話になっております。
以下の件について確認をお願いいたします。

注文番号: A-1031
担当: 鈴木
部署: 営業部
申請金額: 9,000円
納期希望: 2024-04-03
```

ミナは画面を見た。

一件ずつ読めば分かる。だが、いきなりExcelに貼るには表記が揺れすぎている。`氏名` と `名前` と `担当`。`部署` と `部門` と `所属`。`金額` と `申請金額`。金額も `12,000円`、`12000`、`9,000円` が混ざっている。

「これ、最初からExcelに放り込むより、一回テキストとして集めて観察した方がいいですね。」

高村が身を乗り出した。

「お、面白い視点だね？ 詳しく聞かせて。」

「メール本文をそのままExcelに入れると、あとで列に分ける時に地獄です。まずテキストとして集めます。そこで、どんな揺れがあるかを見ます。その日の最後にExcelへまとめる。金曜にCSV化してDBへ持っていく。順番を分けた方が安全です。」

高村は頷いた。

「つまり、Excelには最初からメールを放り込むんじゃなくて、テキストで汚れを見て、整理した後に置きたいってこと？」

「そうです。Excelをゴミ箱にしないために、手前で文字列を見る感じです。」

ミナはホワイトボードに書いた。

```text
困っていること:
  メールの書き方が揺れている

混ざっているもの:
  申請番号
  名前
  部署
  金額
  希望日
  備考

揺れている表現:
  氏名 / 名前 / 担当
  部署 / 部門 / 所属
  金額 / 申請金額
```

黒瀬が言った。

「こうやって書くと、ただメールが多いんじゃなくて、何が混ざっているかが見えるな。」

「はい。まず問題を言葉にします。言葉にできると、リストにできます。」

ミナはメールの一部を指した。

```text
氏名: 山田　太郎
部署: 営業一課
金額: 12,000円
```

「これは文章に見えますけど、項目のリストです。」

```text
氏名
部署
金額
```

「さらに分けると、名前と値のペアです。」

```text
氏名 -> 山田　太郎
部署 -> 営業一課
金額 -> 12,000円
```

「ここまで来ると、Lispでも扱えます。」

高村が画面を見た。

「文章のふりをした申請データか。」

「そうです。人間には文章、DBにはレコード。その間にあるのが、リストです。」

### ミナのメモ

ミナは右下のメモバッファに、ここまでの変換を整理した。

```text
現実:
  営業メールが多く、表記も揺れている。

言語化:
  氏名/名前/担当、部署/部門/所属、金額/申請金額が混ざっている。

リスト化:
  長いメール
    ↓
  必要項目のリスト
    ↓
  名前と値のペア

次へ:
  Emacsで必要行を残し、不要行を消す。
```

## 4. Emacsで文章を選び、残し、消す

ミナは第3章で見た申請メールを、Emacsの一つのバッファに貼り付けた。Excelに入れる前の、仮置き場である。

バッファには、挨拶、署名、備考、必要項目が混ざっていた。

```text
お世話になっております。
申請番号: A-1029
氏名: 山田　太郎
部署: 営業一課
金額: 12,000円
備考: 急ぎ

いつもお世話になっております。
申請番号: A-1030
名前: 佐藤 花子
部門: 営業二課
金額: 12000
備考: 確認してください
```

「このままだと、読むだけで怠ぃです。まず必要な行だけ見たい。」

「検索？」

「検索だけだと、一件ずつ飛ぶだけです。今欲しいのは、必要な行だけを一度に残す作業です。」

### 4.1 必要な行だけ見る `keep-lines`

ミナはまず、必要な項目が含まれていそうな行だけを残すことにした。

「申請番号、氏名、名前、部署、部門、金額。この辺が入ってる行だけ見たいんですよ。」

ミナは式を入力した。

```elisp
(keep-lines "申請番号\\|氏名\\|名前\\|部署\\|部門\\|金額")
```

高村はすぐに眉を寄せた。

「ごめん、その `\\|` って何？」

ミナは、式を分解してホワイトボードに書いた。

```text
申請番号\\|氏名\\|名前\\|部署\\|部門\\|金額
```

「これはEmacsの正規表現です。正規表現って名前は怖いですけど、ここでは『または』の記号だけ覚えれば大丈夫です。`\\|` は『または』という意味です。」

```text
申請番号\\|氏名
  = 「申請番号」または「氏名」を含む行
```

つまり、この式はこう読める。

```text
申請番号 または 氏名 または 名前 または 部署 または 部門 または 金額
を含む行だけ残す。
```

#### 入力例

```text
お世話になっております。
申請番号: A-1029
氏名: 山田　太郎
部署: 営業一課
金額: 12,000円
備考: 急ぎ
```

#### 出力例

```text
申請番号: A-1029
氏名: 山田　太郎
部署: 営業一課
金額: 12,000円
```

ここで、長いメールは「必要そうな行のリスト」になった。

```text
長いメール本文
  ↓ keep-lines
必要そうな行のリスト
```

高村は画面を見た。

「文章が、表になる前の材料になった感じだね。」

「そうです。文章をそのまま表に押し込むんじゃなくて、行のリストにほどいてから扱います。」

### 4.2 不要な行を消す `flush-lines`

必要な行だけを残す方法は分かった。だが、ミナは別のメールを見て、少し首を傾げた。

挨拶や署名が多すぎる。必要な行を残すより、まず邪魔な定型文だけ消した方が早い場合もある。

「お世話になっております、よろしくお願いします、みたいな行が多すぎますね。こういうのは業務的には大事ですけど、データ化するときはノイズです。」

高村が言った。

「人間関係には必要だけど、DBには不要ってことか。」

「そうです。人間には礼儀、機械にはノイズです。」

ミナは不要な行を消す式を書いた。

```elisp
(flush-lines "お世話になっております\\|よろしくお願いいたします")
```

`flush-lines` は、条件に合う行を消す。  
`keep-lines` が残すなら、`flush-lines` は消す。

ただし、問題もある。

```text
備考: よろしくお願いいたします。急ぎです。
```

この行は、備考として意味がある。単純に「よろしくお願いいたします」を含む行を消すと、この備考まで消える可能性がある。

そこで、行全体が定型文だけの場合に限定する。

```elisp
(flush-lines "^よろしくお願いいたします。$")
```

ここで出てくる `^` と `$` は、正規表現の目印である。

```text
^
  行の先頭

$
```

まずは、前後の余計な空白だけを削る。人間が手で消すには地味すぎるが、機械に任せるにはちょうどいい。

```lisp
(string-trim " " " 山田 太郎 ")
```

#### 何をしているか

`string-trim` は、文字列の前後にある指定文字を削る。  
ここでは半角スペースを削っている。

#### 入力例

```lisp

```

#### 出力例

```lisp

```

#### 失敗例

```lisp
(string-trim " " "山田　太郎")
;; => "山田　太郎"
```

全角スペースは半角スペースとは別の文字なので削れない。

#### リファクタリング

全角スペースを半角スペースへ置換してから、前後空白を削る。

```lisp
(defun replace-char (from to string)
  (map 'string
       (lambda (ch)
         (if (char= ch from) to ch))
       string))

(defun normalize-name-space (name)
  (string-trim " "
               (replace-char #\Ideographic_Space #\Space name)))
```

#### 入力例

```lisp
(normalize-name-space " 山田　太郎 ")
```

#### 出力例

```lisp
"山田 太郎"
```

#### 失敗例

```lisp
(normalize-name-space "YAMADA TARO")
;; => "YAMADA TARO"
```

## 6. メール本文を項目のリストへほどく

第4章で必要な行は残った。次は、その行を項目名と値に分ける。

```text
申請番号: A-1029
氏名: 山田 太郎
部署: 営業一課
金額: 12000
希望日: 2024-04-01
```

目標はこれである。

```lisp
(:request-id "A-1029"
 :name "山田 太郎"
 :department "営業一課"
 :amount 12000
 :date "2024-04-01")
```

高村は首を傾げた。

「この `:request-id` とか `:name` って何？」

「plistです。property listの略です。名前と値を交互に並べたメモだと思ってください。」

ミナは紙に書いた。

```text
:request-id  -> "A-1029"
:name        -> "山田 太郎"
:department  -> "営業一課"
:amount      -> 12000
:date        -> "2024-04-01"
```

「メールの行は、最初はただの文字列です。でも、項目名と値に分けると、こういう名前付きのメモにできます。ここまで来ると、Excelの列にも、JSONにも、DBの行にもつなげやすくなります。」

### 6.1 行を名前と値に分ける

ミナはメール本文を見ながら、まず一行だけを指でなぞった。

```text
金額: 12000
```

「この一行は、左が項目名で、右が値です。人間は一瞬で分かる。でも、プログラムには境目を教えないと分かりません。」

境目は `:` だ。だから、まず `:` の位置を探し、左と右に切る。

```lisp
(defun split-pair (line)
  (let ((pos (position #\: line)))
    (when pos
      (list (subseq line 0 pos)
            (subseq line (1+ pos))))))
```

#### 記号の読み方

```text
defun
  関数を作る。

line
  処理したい一行の文字列。

position
  指定した文字が、文字列の何番目にあるかを探す。

#\:
  コロン文字そのものを表す書き方。

subseq
  文字列の一部を切り出す。

0 pos
  先頭からコロンの直前まで。

(1+ pos)
  コロンの次の位置。
```

つまり、この関数は「コロンの左側」と「コロンの右側」に一行を分ける。

#### 入力例

```lisp
(split-pair "金額: 12000")
```

#### 出力例

```lisp
("金額" " 12000")
```

#### 失敗例

```lisp
(split-pair "金額 12000")
;; => NIL
```

`:` がないため、名前と値に分けられない。

#### リファクタリング

全角コロン `：` も扱う。

```lisp
(defun normalize-colon (line)
  (replace-char #\： #\: line))

(defun split-pair* (line)
  (split-pair (normalize-colon line)))
```

#### 入力例

```lisp
(split-pair* "金額： 12000")
```

#### 出力例

```lisp
("金額" " 12000")
```

### 6.2 項目名を内部キーへ変える

```lisp
(defparameter *field-map*
  '(("申請番号" . :request-id)
    ("氏名" . :name)
    ("名前" . :name)
    ("担当" . :name)
    ("部署" . :department)
    ("部門" . :department)
    ("所属" . :department)
    ("金額" . :amount)
    ("申請金額" . :amount)
    ("希望日" . :date)
    ("納期希望" . :date)))

(defun field-key (label)
  (cdr (assoc (string-trim " " label) *field-map* :test #'string=)))
```

#### 記号の読み方

```text
defparameter
  後で使う変数を作る。

*field-map*
  項目名の対応表。
  Common Lispでは、慣習としてグローバル変数を *...* で囲むことが多い。

("氏名" . :name)
  "氏名" という文字を、内部では :name として扱うという対応。

assoc
  対応表の中から、左側が一致するものを探す。

cdr
  見つかった対応の右側を取り出す。

:test #'string=
  文字列を比べるときは string= を使う、という指定。
```

#### 何をしているか

メール内の表記ゆれを、内部の統一キーへ変換する。

```text
氏名     -> :name
名前     -> :name
担当     -> :name
部署     -> :department
部門     -> :department
所属     -> :department
金額     -> :amount
申請金額 -> :amount
```

#### 入力例

```lisp
(field-key "申請金額")
```

#### 出力例

```lisp
:AMOUNT
```

#### 失敗例

```lisp
(field-key "費用")
;; => NIL
```

未知のラベルは変換できない。

### ミナのメモ

ミナは右下のメモバッファに、ここまでの変換を整理した。

```text
現実:
  メール本文の中で、同じ意味の項目が別名で書かれている。

処理:
  1行を「項目名」と「値」に分ける。
  表記ゆれを内部キーへ寄せる。

抽象化:
  "金額: 12000"
    ↓
  ("金額" "12000")
    ↓
  (:amount 12000)

使った道具:
  split-pair
  field-key

次へ:
  複数行をまとめると、1件分の申請データになる。
```

```lisp
(defun naive-split-comma (line)
```

#### 何をしているか

### 7.2 数値変換と失敗

次に、金額欄を整数として扱いたい。

ただし、ここでミナは一つ注意を入れた。

「`nil` だけを返す関数にすると、『値がない』のか『変換に失敗した』のか分かりにくくなります。あと、金額が `0` の時に、うっかりエラー扱いする事故も起こりがちです。」

高村が頷いた。

「0円はあり得るけど、偽っぽく見えるから危ないわけか。」

「そうです。だから、成功したかどうかと、値そのものを分けて返します。」

```lisp
(defun clean-integer-string (s)
  (remove #\, (string-trim " " s)))

(defun parse-integer-result (s)
  (let ((cleaned (clean-integer-string s)))
    (handler-case
        (list :ok t
              :value (parse-integer cleaned)
              :raw s
              :cleaned cleaned)
      (error ()
        (list :ok nil
              :error "not an integer"
              :raw s
              :cleaned cleaned)))))

(defun parse-amount-field (s)
  (parse-integer-result s))
```

#### 記号の読み方

```text
clean-integer-string
  前後の空白を削り、カンマを取り除く。
  例: "18,000" -> "18000"

parse-integer
  文字列を整数に変える。
  例: "18000" -> 18000

handler-case
  エラーが起きたときに、処理全体を止めずに受け止める。

:ok
  変換に成功したかどうか。

:value
  成功した時の整数値。

:error
  失敗した時の理由。

:raw
  元の文字列。

:cleaned
  変換前に掃除した文字列。
```

#### 入力例

```lisp
(parse-amount-field "18,000")
```

#### 出力例

```lisp
(:OK T :VALUE 18000 :RAW "18,000" :CLEANED "18000")
```

#### 0円の例

```lisp
(parse-amount-field "0")
```

#### 出力例

```lisp
(:OK T :VALUE 0 :RAW "0" :CLEANED "0")
```

`0` は正しい値であり、エラーではない。

#### 失敗例

```lisp
(parse-amount-field "12,000円")
```

#### 出力例

```lisp
(:OK NIL
 :ERROR "not an integer"
 :RAW "12,000円"
 :CLEANED "12000円")
```

`円` が残っているため、整数として読めない。

#### リファクタリング

`nil` だけで成功・失敗を表すのではなく、`:ok`、`:value`、`:error` を持つ結果にする。  
これで、「値がない」と「変換に失敗した」と「0という正しい値」を区別できる。

### ミナのメモ

ミナは右下のメモバッファに、ここまでの変換を整理した。

```text

  CSVが表に見えるが、実際には区切り文字を持つ文字列である。

問題:
  カンマを含む値
  引用符
  円付き金額
  数値変換失敗

処理:
  まず壊れる例を見る。
  その後、専用パーサーや検査へ進む。

次へ:
  壊れた値は捨てず、エラー情報として残す。
```

### 9.2 `jget`で入れ子を辿る

高村はコードを見て聞いた。

「これ、Emacsで動くの？」

「Emacs Lispじゃなくて、Common Lispです。EmacsではSLIMEを使って、SBCLに式を送って動かします。」

高村が聞いた。

「また知らない名前が出てきた。SLIMEとSBCLって何？」

「SBCLはCommon Lispを実行する処理系です。つまりLispのエンジンです。SLIMEは、EmacsからそのLispエンジンへコードを送って、結果を受け取るための橋です。」

```lisp
(defun hget (key table &optional default)
  (multiple-value-bind (value foundp)
      (gethash key table)
    (if foundp value default)))

(defun jget (object &rest path)
  (reduce
   (lambda (current key)
     (cond
       ((null current) nil)
       ((hash-table-p current) (hget key current))
       ((and (vectorp current) (integerp key))
        (when (< -1 key (length current))
          (aref current key)))
       (t nil)))
   path
   :initial-value object))
```

#### 記号の読み方

```text
gethash
  hash-tableから値を取り出す。

multiple-value-bind
  gethash が返す「値」と「見つかったかどうか」を別々の名前で受け取る。

&optional default
  値がなかったときの予備の値を指定できる。

&rest path
  残りの引数をまとめて path として受け取る。
  例: "orders" 0 "amount"

reduce
  path を左から順番に処理して、JSONの奥へ進む。

hash-table-p
  今見ているものが hash-table か調べる。

vectorp
  今見ているものが vector か調べる。

aref
  vector の何番目かを取り出す。
```

この関数は、JSONの中を「道順」に沿って歩くための道具である。

#### 入力例

```lisp
(jget json "customer" "name")
```

#### 出力例

```lisp
"Yamada"
```

#### 入力例2

```lisp

```

#### 出力例2

```lisp
"I-09"
```

```lisp
        collect (aref v i)))

    ((null x) nil)

    ((vectorp x) (vector->list x))
    (t (list x))))
```

```text
vector->list

ensure-list
```

```lisp
(defun make-staging-row
    (&key source-file source-branch
          customer-id name email amount registered-at)
  (let* ((clean-name (normalize-name-space name))
         (clean-email (string-trim " " email))
         (amount-result (parse-amount-field amount))
         (amount-ok-p (getf amount-result :ok))
         (clean-amount (when amount-ok-p
                         (getf amount-result :value)))
         (errors '()))
    (unless amount-ok-p
      (push (getf amount-result :error) errors))
    (list
     :source-file source-file
     :source-branch source-branch
     :raw-customer-id customer-id
     :raw-name name
     :raw-email email
     :raw-amount amount
     :raw-registered-at registered-at
     :normalized-customer-id customer-id
     :normalized-name clean-name
     :normalized-email clean-email
     :amount clean-amount
     :registered-at registered-at
     :import-error (when errors
                     (format nil "~{~A~^; ~}" (nreverse errors)))))))
```

#### 何を直したか

以前のように、`maybe-parse-integer` が値そのものか `nil` だけを返す設計だと、`0` のような正しい値を誤ってエラー扱いする危険がある。  
ここでは `parse-amount-field` が `:ok` と `:value` を分けて返すため、`0` も正しい値として扱える。

```lisp
(parse-amount-field "0")
;; => (:OK T :VALUE 0 :RAW "0" :CLEANED "0")
```
